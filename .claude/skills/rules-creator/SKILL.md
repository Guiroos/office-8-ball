# Rules Creator

Create or update `.claude/rules/` files from rule candidates or manual input, following Claude Code best practices for rule organization, format, and path-scoping.

## Step 1: Gather Rule Input

Determine the input source:

**1a — From project-rules-scanner output:**
1. Read the rule candidates from the conversation context. Each candidate follows this format:
   ```
   ### [Candidate Title]
   File: .claude/rules/[target-file].md
   Section: [section name or "new section"]

   [2–5 sentences describing the invariant or convention.]
   ```
2. Parse each candidate into: title, target file, section, and description.

**1b — From manual user input:**
1. Ask the user for the rule topic, which files or paths it applies to, and the convention to enforce.
2. Draft a candidate in the format above.
3. Present the draft and ask for confirmation before proceeding.

## Step 2: Inventory Existing Rules

1. Read all files in `.claude/rules/` recursively.
2. Read `CLAUDE.md` at the project root.
3. Build a list of topics and conventions already documented.
4. For each candidate from Step 1, check for duplication:
   - If a candidate duplicates an existing rule verbatim, discard it and inform the user.
   - If a candidate overlaps partially, flag it and ask the user whether to merge, replace, or skip.

## Step 3: Validate Rule Content

For each candidate, verify:

1. **Tone**: Third-person imperative (e.g., "Use async/await", "Validate input before processing"). Run `python3 .claude/skills/rules-creator/scripts/validate-rule.py --content "[rule text]"` to check.
2. **Length**: 2–5 sentences per rule. Concise and specific.
3. **Actionability**: The rule describes a concrete convention, not a vague preference.
4. **No framework defaults**: Discard rules that restate obvious language/framework behavior.

If validation fails, rewrite the candidate and re-validate. Present the corrected version to the user.

## Step 4: Determine File Organization

For each validated candidate, decide the target file:

**4a — Path-scoping decision:**
Read `references/path-scoping-guide.md` to determine whether the rule needs a `paths` frontmatter field.
- If the rule applies **only** to specific file patterns (e.g., only `src/modules/*/pages/**/*.tsx`), add a `paths` field.
- If the rule applies **globally** to all code, omit the `paths` field.

**4b — File assignment:**
- Group related rules into domain files. Read `references/rule-file-domains.md` for the recommended domain categories.
- If a candidate fits an existing file, assign it there.
- If no existing file fits, propose a new file name following kebab-case convention (e.g., `api-patterns.md`, `form-validation.md`).
- Keep each file under 50 lines to avoid context bloat.

## Step 5: Write Rule Files

For each target file:

**5a — New file:**
1. Read `assets/rule-template.md` for the file structure.
2. Create the file at `.claude/rules/[filename].md` with:
   - Optional `paths` frontmatter (from Step 4a).
   - A top-level `#` heading matching the domain.
   - Rules organized under `##` section headings.
   - Each rule as a bullet point or short paragraph in third-person imperative.

**5b — Existing file:**
1. Read the existing file.
2. Find the correct section (or create a new `##` section if needed).
3. Append the new rule(s) under the appropriate section.
4. Ensure no duplicate content was introduced.
5. Verify the file stays under 50 lines. If it exceeds, split into two domain files.

## Step 6: Final Verification

1. Run `python3 .claude/skills/rules-creator/scripts/validate-rule.py --file .claude/rules/[filename].md` for each written file.
2. Read each written file back to confirm:
   - Frontmatter is valid YAML (if present).
   - Markdown structure is clean (headings, bullets, no orphan text).
   - No duplication with other rule files or CLAUDE.md.
3. Present a summary to the user listing:
   - Files created or modified.
   - Rules added per file.
   - Any candidates that were discarded and why.

## Error Handling

- If `.claude/rules/` directory does not exist, create it before writing files.
- If `scripts/validate-rule.py` fails, perform tone and length validation manually: check for first/second person pronouns, verify 2–5 sentences, and confirm the rule is actionable.
- If a target file exceeds 50 lines after writing, split it into two files by section and update any cross-references.
- If the user provides ambiguous input in manual mode, ask clarifying questions before drafting the candidate. Do not guess.
