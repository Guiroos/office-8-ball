**Step 1: Create proxy.ts at project root**
```typescript
import { getSessionCookie } from "better-auth/cookies";
import { NextResponse, type NextRequest } from "next/server";

const authRequired = Boolean(process.env.DATABASE_URL?.trim());

export default async function proxy(request: NextRequest) {
  if (!authRequired) {
    return NextResponse.next();
  }

  const sessionCookie = getSessionCookie(request);

  if (!sessionCookie) {
    return NextResponse.redirect(new URL("/login", request.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: [
    "/dashboard/:path*",
    "/scoreboard",
    "/times/:path*",
    "/ranking/:path*",
    "/profile/:path*",
    "/settings/:path*",
    "/head-to-head/:path*",
  ],
};
```

Note: `getSessionCookie` only checks for cookie presence — it does NOT validate the session. This mirrors the security model of the previous `withAuth` (which also only checked the cookie signature, not the full session). Actual session validation happens in `getAuthenticatedUser()` on every protected API route. (per RESEARCH.md Pattern 4 security note)

This replaces `src/app/api/auth/[...nextauth]/route.ts`.

**Step 4: Delete src/app/api/auth/[...nextauth]/route.ts**
Delete the file and the `[...nextauth]` directory. This directory must NOT exist alongside `[...all]` — both would match `/api/auth/*` routes and cause conflicts.

**Step 5: Delete src/types/next-auth.d.ts**
This file augmented `next-auth` types for the JWT session. It is no longer needed — better-auth types are derived from `auth.$Infer.Session`. Deleting this file removes the last `next-auth` reference from the type system. (per D-13)

After deletion, TypeScript will no longer have the session type augmentation from next-auth. The `SessionUser` type in `src/lib/types.ts` is unchanged — it was always independent of next-auth types.
