# Phase 2 — UI Design Contract: Scoreboard Reactivation & Match Recording

> Visual and interaction contract for Phase 2. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Executive Summary

Phase 2 requires **minimal visual changes**. The dashboard UI structure remains unchanged; the primary work is removing hardcoded team constants (`frontend`/`backend`) and connecting to dynamic team APIs. The scoreboard reactivation is an API-layer change that will be consumed by existing dashboard components through refactored data hooks.

**Design contract impact:**
- No new components
- No color or typography changes
- No spacing scale changes
- Dashboard continues to display two-team scoreboard pattern, but teams are now fetched dynamically
- Match recording form remains identical, now validates against user's actual team memberships

## Interaction Contract

### Dashboard Team Display

**Current (Phase 1 hardcoded):**
```tsx
const TEAMS = [
  { id: "frontend", name: "frontend", displayName: "Frontend", roster: "Gui + Jean", ... },
  { id: "backend", name: "backend", displayName: "Backend", roster: "Adair + Richard", ... },
];
```

**Phase 2 (dynamic):**
- Fetch from `/api/teams` endpoint in `useTeamsData()` hook
- Type: `TeamRecord[]` (from `src/lib/types.ts`)
- Update `TeamScoreCard` to accept dynamic `team` object instead of hardcoded lookup
- Map teams to scoreboard data via `id` matching

**Interaction behavior (unchanged):**
- Display two score cards side-by-side (lg: grid-cols-2)
- Each card shows team name, roster, wins, and optional "Provocação" note
- Leader badge appears when `isLeader={true}`
- "Registrar vitória" button remains full-width, styled by team variant
- No layout changes; same responsive grid

### Match Registration

**Current state:**
- User selects winning team from hardcoded `TEAMS` constant
- Form validates `winnerTeamId` exists in constants
- Submits with `teamAId: frontend, teamBId: backend` (hardcoded match pair)

**Phase 2 (dynamic teams):**
- Match recording still submits a single winning team; implicit opponent is the "other team" (out of scope for Phase 2; Phase 4 adds multi-team matching)
- Validation on `/api/matches` confirms:
  1. `teamAId`, `teamBId` exist in database
  2. Both teams are active
  3. User is member of at least one team
  4. `winnerTeamId` is one of `teamAId` or `teamBId`
- No UI changes to form; same error messages in Portuguese via `sonner` toast

### Empty State (Future)

**Not in Phase 2 scope.** If user has no teams:
- Dashboard should display empty state (to be designed in Phase 4)
- Currently: Scoreboard component will render with empty `teams` array
- Copy: "Você ainda não criou nenhum time. [Link to create team]" (deferred to Phase 4)

## Scoreboard Data Shape (API Contract)

**Current shape (Phase 1):**
```typescript
type ScoreboardData = {
  teams: Array<{ id: string; displayName: string; wins: number }>;
  leaderTeamId: string | null;
  leadBy: number;
  totalMatches: number;
  currentStreak: { teamId: string; teamName: string; count: number } | null;
};
```

**Phase 2 (reactivated for dynamic teams):**
The shape remains the same structure. However:
- `teams` array now contains ALL user's teams (not just `frontend`/`backend`)
- Wins/losses computed from match history for each team dynamically
- If user has > 2 teams, scoreboard will eventually render > 2 cards (future phase)
- If user has 0 or 1 team, scoreboard renders empty or single card (empty state handling deferred to Phase 4)

**For Phase 2 MVP:** Assume users can see up to N teams. Dashboard grid remains `lg:grid-cols-2`; if more than 2, they wrap to next rows.

## Team Color Mapping — Design Decision

**Phase 1 hardcoded:**
```tsx
const TEAMS = [
  { id: "frontend", accent: "var(--team-alpha)", ... },
  { id: "backend", accent: "var(--team-beta)", ... },
];
```

**Phase 2 dynamic:**
User-created teams do not have hardcoded color tokens. Solution:

1. **Option A (Chosen):** Reserve `team-alpha` and `team-beta` for the first two teams in the user's team list (alphabetically or by creation order). Implement in Phase 4 stats module when multi-team display is finalized.

2. **Option B:** Extend tokens to support `team-gamma`, `team-delta`, etc. as user creates more teams. (Deferred to Phase 4.)

3. **For Phase 2 MVP:** Continue using `team-alpha` and `team-beta` in UI classes. These become cosmetic; actual team styling will be refined in Phase 4 when full team list is visible.

**Implementation:** Dashboard will continue to render `bg-team-alpha-soft` and `bg-team-beta-soft` for the first two teams. If > 2 teams exist, reuse colors or use semantic token variants (e.g., `bg-surface-emphasis`). Phase 4 defines the color rotation strategy.
